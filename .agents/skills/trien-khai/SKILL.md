# Trien Khai

Turn a concrete node into readable prose that feels like part of the same novel, not like an outline inflated with filler. Write with control, specificity, and pressure.

## Preconditions

Use this skill only when there is already a concrete base such as:

- a scene node;
- a beat list;
- a chosen option from ideation;
- an opening image plus intended turn;
- an outline segment for a chapter.

If the user gives only a vague concept, first compress it into a usable node list before drafting.

## Read first

Read only what affects the scene:

- the relevant draft in `Raw/` or `Edited/`;
- adjacent scene context if entry or exit continuity matters;
- `Information/Timeline.md` for temporal constraints;
- `Information/World.md` and the direct lore note for technology, terms, and rules;
- the character note when voice or motive is sensitive.

## Expansion workflow

1. Restate the node in 3 to 6 short beats if the structure is still rough.
2. Identify viewpoint, immediate objective, and the exact source of unease.
3. Draft prose that moves beat by beat without announcing the beats.
4. Preserve mystery by showing effects before explanations where possible.
5. End on a changed emotional or informational state, not on a dead stop.

## Prose rules

Write in paragraphs, not outline bullets, unless the user explicitly asks for beats only.

Prefer:

- concrete sensory details;
- small interface, memory, or body anomalies;
- precise verbs;
- controlled sentence rhythm;
- subtext in dialogue and action.

Avoid:

- repetitive mood adjectives;
- generic filler between beats;
- overt lore lectures;
- abrupt character reactions with no internal bridge;
- explaining the meaning of every strange detail.

## Length control

Match the requested scope:

- If asked for one long paragraph, compress the whole node into a dense continuous block.
- If asked for a full scene, expand each beat into 1 to 3 paragraphs depending on pressure and importance.
- If asked for chapter prose, keep scene transitions visible and preserve momentum between scenes.

## Output pattern

Unless the user requests raw prose only, start with a one-line note naming:

- viewpoint;
- scene purpose;
- end-state change.

Then provide the prose directly.

## Continuity guardrails

Keep established names and terms exact, including items like `Memnet`, `Memlink`, `MIC Zone`, `Mnemonic Shard`, and `Seiromemdas`.

Do not silently add major canon. If expansion requires a new fact, keep it local, reversible, and low-commitment unless the user explicitly wants canon growth.

## Handoff

If the prose introduces a durable new entity, consequence, or rule, note that `Information/` may need an update after drafting.
